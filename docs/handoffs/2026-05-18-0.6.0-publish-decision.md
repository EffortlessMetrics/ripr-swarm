## Exact Release Execution Steps

Historical runbook. The `v0.6.0` tag, GitHub Release, server assets, crates.io
package, VS Code Marketplace package, and Open VSX package already exist. Do
not run these steps again unless a maintainer explicitly opens a release
recovery task.

### 1. Prepare the local release checkout

```bash
git fetch origin main --tags
git checkout main
git pull --ff-only origin main
git rev-parse HEAD
git status --short
```

Expected `HEAD` should be the latest `origin/main` commit that contains this
publish-decision handoff and no later code, release-surface, policy,
generated-evidence, or dependency changes. At the time #1216 merged, that
commit was:

```text
7f7e5a1c66258ce1f775daad9f2248b9d765767c
```

If a later docs-only correction to this handoff lands, the newer correction
commit supersedes the hash above. Abort if `HEAD` contains any later code,
release-surface, policy, generated-evidence, or dependency change unless the
maintainer explicitly accepts the newer release base and reruns the required
proof.

### 2. Confirm the final local gates

```bash
cargo xtask check-generated-clean
cargo xtask release-readiness --version 0.6.0
cargo publish -p ripr --dry-run
npm --prefix editors/vscode run package
```

Abort if any required release-readiness check fails, if generated residue is
tracked, or if the package/VSIX dry-run no longer passes.

### 3. Create and push the release tag

```bash
git tag v0.6.0
git push origin v0.6.0
```

Abort if `v0.6.0` already exists locally or remotely.

### 4. Build and attach server assets

```bash
gh workflow run release-server-binaries.yml -f version=0.6.0
```

Wait for the workflow to pass. Then verify the GitHub Release assets:

```bash
gh release view v0.6.0 --repo EffortlessMetrics/ripr --json name,tagName,publishedAt,assets,url,isDraft,isPrerelease
gh release download v0.6.0 --repo EffortlessMetrics/ripr --pattern 'ripr-server-v0.6.0-x86_64-pc-windows-msvc.zip' --pattern 'ripr-server-manifest-v0.6.0.json' --dir target/ripr/release-smoke --clobber
```

Abort crates.io and marketplace publication if the release lacks the VSIX,
manifest, checksums, or per-target server archives required by
[Release binaries](../RELEASE_BINARIES.md).

### 5. Publish crates.io

```bash
cargo login
cargo publish -p ripr
```

If Cargo times out while polling after upload, check crates.io manually before
retrying. Do not run a second publish until the registry state is known.

### 6. Verify crates.io install

```bash
cargo install ripr --version 0.6.0 --locked --root target/ripr/install-smoke-cratesio --force
target/ripr/install-smoke-cratesio/bin/ripr --version
target/ripr/install-smoke-cratesio/bin/ripr first-pr --help
target/ripr/install-smoke-cratesio/bin/ripr doctor
target/ripr/install-smoke-cratesio/bin/ripr pilot --root fixtures/boundary_gap/input --out target/ripr/install-smoke-cratesio/pilot
target/ripr/install-smoke-cratesio/bin/ripr outcome --before fixtures/boundary_gap/calibration/before-targeted-test.repo-exposure.json --after fixtures/boundary_gap/calibration/after-targeted-test.repo-exposure.json
```

### 7. Publish VS Code Marketplace and Open VSX

After server assets are present and verified:

```bash
gh workflow run publish-extension.yml --field version=0.6.0
```

Manual fallback:

```bash
cd editors/vscode
npm ci
npm run compile
npm run package
npx @vscode/vsce publish --packagePath dist/ripr-0.6.0.vsix --pat "$VSCE_PAT"
npx ovsx publish dist/ripr-0.6.0.vsix -p "$OVSX_PAT" --skip-duplicate
```

### 8. Post-publish smoke

Verify:

Only refresh manual marketplace badge counts after marketplace publication and
metrics review. Do not hand-edit repo-scoped generated badge endpoint JSON.

## Abort Conditions

Abort release execution if any of these happen:

- local `HEAD` is not the accepted decision-packet base and no refreshed proof
  or docs-only correction exists;
- required release-readiness check fails;
- `cargo publish -p ripr --dry-run` fails;
- VSIX packaging fails;
- generated badge endpoints or other generated residue appear as tracked diffs;
- GitHub Release asset generation fails or assets are missing;
- crates.io already contains `ripr` 0.6.0 from a previous publish attempt;
- secrets or tokens are missing, expired, or exposed;
- public copy would imply runtime mutation proof, generated tests, provider
  execution, default blocking, source edits, or preview-language gate authority.

## Rollback / Fix-Forward Limits

- After `git push origin v0.6.0`, do not retag unless maintainers explicitly
  approve an exceptional release recovery.

  the release body must record any user-visible recovery.
- Marketplace packages should be fixed forward with a new version if a broken
